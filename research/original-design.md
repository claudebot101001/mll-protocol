# Mutual Liquidity Lock (MLL) Protocol

> 一种将抽象承诺转化为可执行金融约束的去信任化协议

---

## 核心洞察

**承诺的本质问题**：口头承诺没有成本，因此没有可信度。

**MLL的解决方案**：将承诺 tokenize 为一个双边流动性资产——

- 双方持续向资产池注入流动性（定期投入）
- 流动性提取需要双方共同签名（2-of-2 多签）
- 单方违约导致流动性损失（自动惩罚）
- 双方冲突导致流动性永久冻结（MAD 威慑）

```
承诺可信度 = f(累积投入, 退出成本, 时间)
```

当退出成本足够高，承诺就从"说说而已"变成"用真金白银担保的契约"。

## 协议架构

### 1. 基础结构

```
┌─────────────────────────────────────────────────┐
│              Mutual Liquidity Pool              │
│                                                 │
│   ┌─────────┐                   ┌─────────┐    │
│   │ Party A │ ──── 2-of-2 ───── │ Party B │    │
│   └────┬────┘      MultiSig     └────┬────┘    │
│        │                              │         │
│        ▼                              ▼         │
│   Monthly          Shared           Monthly     │
│   Deposit ───────► Reserve ◄─────── Deposit    │
│                       │                         │
│                       ▼                         │
│              DeFi Yield Generation              │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 2. 参与方权利与义务

| 权利 | 义务 |
|------|------|
| 共享 DeFi 收益 | 按期投入约定金额 |
| 和平退出时取回本金 | 维护多签钱包安全 |
| 发起俄罗斯转盘 | 承担终局风险 |

### 3. 资金流动规则

**流入**：任一方可随时单方存入（无需对方签名）

**流出**：仅限以下三种情况——
- 双方签名同意提取
- 失血机制自动转移（违约惩罚）
- DeFi 收益自动分配

## 四大核心机制

### 机制一：持续投入信号（Continuous Commitment Signal）

**设计**：双方约定投入计划，按期向共同池注资

**参数建议**：
- 投入频率：每月
- 投入比例：可选档位（5% / 10% / 20% 月收入）
- 递增计划：可选（如第一年 5%，第二年 10%）

**博弈论基础**：源自 Spence (1974) 信号理论——持续的高成本行为是可信的承诺信号

**实践意义**：
- 每月投入 = 可观测的"心跳"信号
- 停止投入 = 立即可见的承诺衰减
- 无需猜测对方态度，行为本身就是信息

### 机制二：失血惩罚（Bleeding Penalty）

**设计**：一方未按期投入时，资金自动以固定速率流向守约方

**参数建议**：
- 速率：0.5% / 天（约 15% / 月）
- 触发条件：超过约定日期未投入
- 终止条件：补缴完成或池子清空

**博弈论基础**：将消极不作为转化为主动可见的成本

**数学分析**：
```
设池中总额为 P，失血速率为 r = 0.5%/天

30天后守约方可提取：P × (1 - 0.995³⁰) ≈ P × 14%
60天后守约方可提取：P × (1 - 0.995⁶⁰) ≈ P × 26%
180天后守约方可提取：P × (1 - 0.995¹⁸⁰) ≈ P × 59%
```

**设计考量**：
- 足够慢：允许从意外情况（失联、事故）中恢复
- 足够快：让消极抵抗有明确代价
- 不可逆：无法通过拖延来博弈

### 机制三：俄罗斯转盘（Graduated Destruction）

**设计**：任一方可发起"核按钮"，但销毁是概率性的

**机制细节**：
1. 合约创建时，链上随机生成隐藏数 N ∈ [1, 6]
2. 每次发起调用，系统判定是否 = N
3. 若命中，触发终局；若未命中，记录调用次数
4. 调用次数由双方共享（A 调用后，B 的下一次调用是第 2 次）
5. 每次调用后进入 10 天冷静期

**概率分布**：

| 调用次数 | 本次触发概率 | 累积触发概率 |
|---------|-------------|-------------|
| 1 | 16.7% | 16.7% |
| 2 | 20.0% | 33.3% |
| 3 | 25.0% | 50.0% |
| 4 | 33.3% | 66.7% |
| 5 | 50.0% | 83.3% |
| 6 | 100% | 100% |

**博弈论基础**：
- Schelling (1960) 边缘政策：不确定的威胁比确定的威胁更可信
- 保留谈判空间：每次调用后都有 10 天冷静期
- 升级成本递增：调用次数越多，下次触发概率越高

### 机制四：建设性终局（Constructive Endgame）

**三种终局状态**：

```
┌─────────────────────────────────────────────────────────┐
│                      终局状态                            │
├─────────────┬─────────────────┬─────────────────────────┤
│   和平分手   │    冻结信托      │      完全销毁           │
│  (Peaceful)  │ (Frozen Trust)  │    (Total Loss)        │
├─────────────┼─────────────────┼─────────────────────────┤
│ 双方签名同意 │ 俄罗斯转盘触发  │      不采用             │
│ 按约定比例   │ 或长期僵局      │                        │
│ 分割本金     │                 │                        │
├─────────────┼─────────────────┼─────────────────────────┤
│ 本金：各取   │ 本金：永久锁定  │        N/A             │
│ 收益：各取   │ 收益：持续平分  │                        │
└─────────────┴─────────────────┴─────────────────────────┘
```

**为什么选择"冻结"而非"销毁"**：
- 销毁 = 纯粹的相互伤害，无建设性
- 冻结 = 保留长期和解可能性
- 收益分配 = 即使冲突，双方仍有被动收入纽带

## 参数协商机制

### 问题：如何在收入不透明时建立公平的投入比例？

**方案：TCP 式探测算法**

借鉴 TCP 拥塞控制的思路，通过行为而非声明来发现双方的真实承受能力：

```
Phase 1: 指数探测期
─────────────────────
  Month 1: 双方各投入 X₀（协商的起始值）
  Month 2: 双方各投入 X₀ × α
  Month 3: 双方各投入 X₀ × α²
  ...
  直到某方发出"压力信号"

Phase 2: 回退与线性逼近
─────────────────────
  收到压力信号后，回退至上月金额
  此后每月增长率降为 β（如 1.05）
  
Phase 3: 稳定期
─────────────────────
  连续 3 个月无压力信号 → 锁定为基数
  基数每年可重新协商一次
```

**参数建议**：
- 起始值 X₀：双方协商，如 1000 元
- 指数增长率 α：1.5 - 2.0
- 线性增长率 β：1.05 - 1.10
- 压力信号：明确的协议内操作，非口头声明

**机制优势**：
- 无需披露收入
- 通过行为揭示真实能力
- 双方阈值不同时，取较低者
- "故意低报"会在后续合作中暴露

### 分层承诺等级

| 等级 | 投入比例 | 适用场景 | 信号强度 |
|------|---------|---------|---------|
| Tier 1 | 5% | 试探期、低收入群体 | ★☆☆ |
| Tier 2 | 10% | 标准承诺 | ★★☆ |
| Tier 3 | 20% | 高承诺需求、高净值人群 | ★★★ |

**规则**：双方必须选择同一等级。选择本身就是信号。

## 应用场景

### 场景一：亲密关系

**适用条件**：
- 双方希望用可量化方式表达长期承诺
- 法律婚姻不可用或不适用
- 一方或双方有过被背叛经历

**配置建议**：
- 投入比例：Tier 2 或 Tier 3
- 俄罗斯转盘：启用
- 终局模式：冻结信托

### 场景二：合伙创业

**适用条件**：
- 合伙人贡献异质（资金 vs 技术 vs 资源）
- 担心中途退出或搭便车
- 需要持续投入而非一次性承诺

**配置建议**：
- 投入方式：按股权比例或约定比例
- 失血机制：可调整为更快（如 1%/天）
- 终局模式：可设置里程碑解锁

### 场景三：长期服务合约

**适用条件**：
- 甲方担心乙方跑路
- 乙方担心甲方拖款
- 跨境合作法律执行成本高

**配置建议**：
- 双方各锁定合同总额的 X%
- 里程碑完成后释放对应比例
- 争议时启用俄罗斯转盘

### 场景四：学术合作

**适用条件**：
- 长周期项目（1年+）
- 署名权、贡献比例争议风险
- 一方可能中途放弃

**配置建议**：
- 投入金额：象征性（如各 1000 元/月）
- 核心约束：持续投入信号
- 终局意义：声誉成本而非财务成本

## 理论基础

### 博弈论渊源

| 概念 | 来源 | 在 MLL 中的应用 |
|------|------|----------------|
| 承诺装置 | Schelling (1960) | 金融质押使承诺可信 |
| 相互保证毁灭 | 冷战威慑理论 | 冻结信托作为终极威慑 |
| 人质交换 | Raub & Keren (1993) | 双向质押作为相互人质 |
| 边缘政策 | Schelling (1960) | 俄罗斯转盘的概率升级 |
| 信号理论 | Spence (1974) | 持续投入作为承诺信号 |
| 拥塞控制 | TCP/IP | 参数协商的探测机制 |

### 金融原语视角

MLL 可以被理解为一种新的**金融原语（Financial Primitive）**：

```
传统金融原语：
  - 存款（单方存入，单方提取）
  - 贷款（单方借出，约定偿还）
  - 保险（单方付费，条件触发赔付）
  - 期权（单方购买，条件触发执行）

MLL 原语：
  - 双方存入
  - 双方持续追加
  - 提取需双方同意
  - 违约自动惩罚
  - 冲突导致流动性冻结
```

这是一种**双边流动性锁定（Bilateral Liquidity Lock）**，现有金融体系中没有直接对应物。

## 开放问题

### 机制层面

1. **收入验证**：链上无法验证链下收入，如何防止故意低报？
2. **多方扩展**：2-of-2 如何推广到 M-of-N？投票权如何设计？
3. **里程碑触发**：长期项目如何定义"持续投入"的节点？
4. **紧急退出**：死人开关如何设计？60天？90天？

### 实践层面

5. **法律地位**：智能合约承诺在不同司法管辖区的效力？
6. **税务处理**：锁定资产的税务性质？收益的税务性质？
7. **市场接受度**：愿意接受这种约束的用户有多少？

### 伦理层面

8. **权力不对等**：收入差距大时，等比例投入是否公平？
9. **退出自由**：过高的退出成本是否限制了正当的关系终止？
10. **心理影响**：将亲密关系金融化是否会改变关系本质？
